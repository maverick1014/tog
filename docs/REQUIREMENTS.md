# 主恩堂（TOG / Tabernacle of Grace）教会管理系统 · 需求与用户故事

> 本文档由代码审计生成，严格对应仓库中**已实现**的行为。凡 API 支持但 UI 未暴露、或
> 尚未实现的能力，均在对应章节以「⚠ 缺口」标注，不做臆测。技术栈：Next.js 15（App
> Router）+ React 19 + Supabase，部署于 Cloudflare Workers（OpenNext）。全部界面为简体
> 中文、仅浅色主题、移动优先。API 为单一 catch-all 路由
> `apps/web/src/app/api/[...path]/route.ts`；鉴权为 HMAC 签名 Cookie
> （`apps/web/src/lib/server/auth.ts`）。

---

## 1. 系统概述

主恩堂教会管理系统是一套面向教牧团队的内部后台，用于牧养全流程管理：成员目录与个人档
案、小组编制与「铁三角」带领团队及每周出席、聚会点名、奉献记录与汇总、培训课程（场次 /
报名审核 / 核对名单）、以及「四十天一对一守望」门训接棒体系。系统以「登录账户 →
关联成员」的双层身份模型运作：**权限角色**（超级管理员 / 管理员 / 同工 / 只读）决定能在
应用中「做什么」，而成员在教会 / 小组中的**身份**（牧师 / 小组长 / … / 新成员）则由小组
管理页逐人分配、只读展示。所有写操作在服务端经统一鉴权闸门校验，客户端亦按会话角色隐藏
无权操作的入口；唯一的公开页面是带领者凭专属 token 访问的「每日守望填写表单」
（`/d/[token]`）。

## 2. 角色与权限矩阵

权限来源：服务端闸门 `route.ts`（权威）+ 客户端 `lib/perms.ts` 的 `can(role)`（仅用于隐
藏 / 禁用 UI）。矩阵反映**实际实现**：

| 能力 | 超级管理员 super_admin | 管理员 admin | 同工 coworker | 只读 readonly |
|---|:---:|:---:|:---:|:---:|
| 登录（账户须 active） | ✅ | ✅ | ✅ | ✅ |
| 查看数据（成员/小组/聚会/奉献/培训/门训/仪表盘） | ✅ | ✅ | ✅ | ✅ |
| 新增 · 编辑（牧养数据写操作，非 GET 且非 DELETE） | ✅ | ✅ | ✅ | ❌ |
| 删除（DELETE） | ✅ | ✅ | ❌ | ❌ |
| 账户管理（`/accounts*` 的**读与写**：列表、新建、编辑、删除、重设密码） | ✅ | ❌ | ❌ | ❌ |
| 自助修改我的密码（`/auth/password`，须验证当前密码） | ✅ | ✅ | ✅ | ✅ |
| 公开门训表单（`/d/[token]` 及 `/api/discipleship/form/*`，无需登录） | 公开（任何持 token 者） | 公开 | 公开 | 公开 |

服务端闸门要点（`route.ts` L47–65）：

- `r0 === 'auth'` 与 `discipleship/form/*` 为公开路径，其余一律要求有效会话，否则 `401 未登录`。
- `accounts*` 的**任意方法（含 GET）** 仅 `super_admin`，否则 `403 仅超级管理员可管理登录账户`（读也受限，避免账户邮箱 / 角色 / 登录历史泄露）。
- 非 GET 时：`readonly` 一律 `403 只读账户无法修改`；`DELETE` 仅 `super_admin` / `admin`，否则 `403 当前角色无删除权限`。
- 密码：PBKDF2-HMAC-SHA256 加盐存储，最少 8 位；会话为 HMAC-SHA256 签名 Cookie（`tog_session`，HttpOnly / Secure / SameSite=Lax，7 天有效）。

## 3. 功能需求

> 约定：C=Create、R=Read、U=Update、D=Delete。「写」= 同工及以上；「删除」= 管理员及以上。
> 所有破坏性操作均须经共享 `useConfirm()`（`danger: true`）确认对话框，禁用原生
> `window.confirm`。

### 3.1 仪表盘 `/`（仪表盘）

- **R 只读概览**：KPI 卡（成员总数、在册成员、即将聚会、本月奉献 RM、门训进行中对数）、
  成员身份分布条形图、近八周奉献趋势 sparkline、即将到来的聚会（前 4）、门训进度关注（进
  度最低前 5）。
- 无任何写操作；所有登录角色可见。数据来自 `/members`、`/events`、`/donations`、
  `/discipleship/programs` 及首个计划的 `/overview`。

### 3.2 成员目录 `/members` 与成员详情 `/members/[id]`（成员目录）

- **C 新增成员**（列表页「＋ 新增成员」，仅 `perms.write` 显示）：姓名（必填）、电话、
  状态（在册 / 停止聚会）、加入日期、邮箱；`church_role` 固定为 `member`。身份 / 小组
  **不在此设定**（提示引导至小组管理）。
- **R 列表**：身份筛选 chip（含各身份计数 + 未分组）、姓名搜索、导出 Excel（`exportRows`，
  懒加载 SheetJS）。桌面表格 / 移动列表瓦片两种呈现，点击进入详情。
- **R 详情**：头像、身份徽章、事实卡（邮箱 / 电话 / 性别 / 所属小组 / 状态 / 加入日期 /
  生日 / 家庭）、**个人培训档案**（来自 `/members/[id]/trainings`）、**四十天守望**配对列
  表（点击打开共享 `PairProgressModal` 进度弹窗）。
- **U 编辑资料**（详情页「编辑资料」，仅 `perms.write`）：姓名 / 英文名 / 电话 / 邮箱 /
  性别 / 生日 / 加入日期 / 状态 / 备注。身份与小组不在此改。
- **U 头像上传**（「上传头像 / 更换头像」，仅 `perms.write`）：`POST /members/[id]/avatar`
  （multipart），存入 Supabase `avatars` bucket。
- **D 删除成员**（详情页「删除」，仅 `perms.delete`）：确认对话框提示「培训、配对与出席记录将一并移除，且不可恢复」，删除后回到目录。
- ⚠ 说明：无法在 UI 中设置 `church_role = pastor`（牧师身份只能经数据库设定）。

### 3.3 小组管理 `/groups`（小组管理 · 身份分配中心）

- **C 新增小组**（「＋ 新增小组」，仅 `perms.write`）：名称（必填）、简介 / 聚会时间。
- **R**：小组 chip 切换（含人数），左栏「小组资料 · 带领团队（铁三角）」，右栏「组员分配」。
- **U 编辑小组**（「保存设定」，仅 `perms.write`）：名称、简介。
- **D 删除小组**（「删除小组」，仅 `perms.delete`）：确认「组员将变为未分组」。
- **组员分配（对成员的 U）**：
  - **添加成员入组**（下拉 + 「＋ 添加成员」，仅 `perms.write`）：`PATCH /members/[id]`
    设 `group_id` 并置身份为 `new_member`。
  - **移出小组**（「移除」，仅 `perms.write`）：确认对话框；清除 `group_id` 与 `group_position`。
  - **调整在组身份**（下拉，仅 `perms.write`）：六个职位可选。
- **特别规则**：
  - **铁三角每职位仅一人**：小组长 / 副组长 / 实习组长每组各一名；指派新领袖时，系统自动
    把原任者降为核心成员（`changePosition` 中 auto-demote）。
  - **领导须先为核心成员**：只有核心成员（或已在领导位）可晋升为三种组长职位
    （`canPromoteToLeadership`），下拉中不满足者对应选项 `disabled`。
- **每周出席（子模块 WeeklyAttendance）**：
  - **C 添加一周**（「＋ 添加一周」，仅 `perms.write`）：`POST /groups/[id]/meetings`，以
    「上周 +7 天」或今天生成日期，展示为「第 N 周」。
  - **U 勾选出席**（复选框，仅 `perms.write`）：切换 `present` / `absent`，
    `POST /groups/meetings/[meetingId]/attendance`。只读用户复选框 `disabled`。
  - **D 删除本周**（表头「✕」，仅 `perms.delete`）：确认对话框后
    `DELETE /groups/meetings/[meetingId]`。
  - **R 导出**：`exportMatrix` 导出成员 × 周次矩阵 + 出席次数。
- ⚠ 小提示：空态文案写「点『＋ 添加本周』」，实际按钮为「＋ 添加一周」（文案不一致，非功能缺陷）。会议 `note` 字段 API 支持但 UI 未采集。

### 3.4 聚会与出席 `/events`（聚会与出席）

- **C 新增聚会**（「＋ 新增聚会」，仅 `perms.write`）：标题（必填）、类型（崇拜 / 聚会 /
  祷告会 / 团契 / 其他）、地点、开始时间（必填）。
- **R**：卡片网格，按开始时间倒序，标注「即将 / 已结束」。
- **U 编辑聚会**（卡片「编辑」，仅 `perms.write`）：同新增字段。
- **D 删除聚会**（卡片「删除」，仅 `perms.delete`）：确认「及其出席记录」。
- **点名（U 出席）**（卡片「点名」，所有登录角色可打开弹窗）：对**在册**成员逐一标记
  出席 / 请假 / 缺席，一次保存 `POST /events/[id]/attendance`（upsert）。
  - ⚠ 客户端未按角色隐藏「点名」按钮与保存动作；只读用户可打开点名弹窗并点「保存点名」，
    但服务端 `POST` 会以 `403 只读账户无法修改` 拒绝（保存报错 toast）。属 G2 客户端 UX 未
    完全对齐（按钮总会 403），见 §6。

### 3.5 奉献管理 `/donations`（奉献管理）

- **C 录入奉献**（「＋ 录入奉献」，仅 `perms.write`）：奉献人（可匿名）、日期、类别
  （十一奉献 / 主日奉献 / 建堂 / 宣教 / 感恩）、方式、金额（RM，须 > 0），币种固定 MYR。
- **R**：顶部四类汇总卡（来自 `/donations/summary`），明细表格（`table.stack` 移动堆叠），
  导出 Excel。
- **U 编辑奉献**（行内「编辑」，仅 `perms.write`）：同录入字段。
- **D 删除奉献**（行内「删除」，仅 `perms.delete`）：确认对话框注明日期 / 奉献人 / 类别。
- 操作列仅在 `perms.write || perms.delete` 时渲染。

### 3.6 培训课程 `/trainings` 与详情 `/trainings/[id]`（培训课程）

- **C 新增课程**（「＋ 新增课程」，仅 `perms.write`，`TrainingModal`）：名称（必填）、类别、
  场次数、讲师（成员）、起止日期、是否开放报名。
- **R 列表**：按「进行中 / 已结束·已截止」分组（结束依 `ends_on < now` 或 `is_enrollable=false`）。
- **R 详情**：课程信息、课程场次列表、报名审核、核对名单矩阵。
- **U 编辑课程**（详情「编辑课程」，仅 `perms.write`，`TrainingModal`）。
- **D 删除课程**（列表卡片或详情「删除」，仅 `perms.delete`）：确认「报名与名单记录将一并移除」。
- **场次**：
  - **C 加场次**（详情「＋ 加场次」，仅 `perms.write`）：第几课（自动取下一号）、标题、时间、地点。
  - ⚠ **缺口（G1）**：API 支持 `PATCH` / `DELETE /trainings/sessions/[id]`，但 UI **未提供场次的编辑 / 删除**入口。
- **报名**：
  - **C 加入报名**（「加入报名成员…」下拉，仅 `perms.write`）：`POST /trainings/[id]/enroll`，
    默认状态 `pending`。
  - **U 通过报名**（「通过」，仅 `perms.write` 且状态为 `pending`）：`PATCH` 置
    `approved`。
  - ⚠ **缺口（G1）**：API 支持将报名改为 `in_progress` / `completed`（自动写
    `completed_at` 与 `progress=100`）/ `dropped`，及 `DELETE /trainings/enrollments/[id]`；
    但 UI **仅暴露 pending→approved**，无「拒绝 / 移除报名 / 标记完成 / 退出」入口。
- **核对名单（U 出席）**（点击方格切换 ✓，仅 `perms.write`）：
  `POST /trainings/sessions/[id]/attendance`；名单仅含 `approved` / `in_progress` /
  `completed` 报名。**R 导出名单**（`exportMatrix`）。只读用户方格不可点。

### 3.7 四十天守望 `/discipleship`（门训接棒）

- 依赖首个 `discipleship_programs` 记录（`programId`）。⚠ **缺口**：若无计划，页面提示
  「请先在数据库中创建 discipleship_programs 记录」——**UI 无法创建 / 编辑 / 删除计划**
  （API 有 `GET/POST /discipleship/programs`，但页面未暴露新建计划入口）。
- **C 新增配对**（「＋ 新增配对」，仅 `perms.write`，须已有 programId）：选带领者 + 被带领
  者（已在配对中的被带领者不显示；不能自配）。系统按带领者已有配对自动串接 `parent_pair_id`
  形成接棒链。
- **R**：
  - 「培育链 · 接棒图」世代培育树（SVG 连接线），可按「在训 / 已出师 / 待开始」筛选，
    「在训」视图支持 **⛶ 全屏**。
  - 「牧者总览」表格 / 移动瓦片：配对（被带领 ← 带领）、进度条、状态，「进度」打开共享
    `PairProgressModal`，「🔗 表单」新开公开填写页。
- **U 进度**：仅通过公开门训表单 `/d/[token]`（见 §3.9）或 `PairProgressModal` 中复制 /
  打开表单链接来记录；⚠ **应用内无直接编辑每日进度的写控件**，进度录入完全依赖表单页
  `POST /discipleship/form/[token]/progress`（upsert）。
- **D 删除配对**（「删除」，仅 `perms.delete`）：确认「守望进度记录将一并移除」，
  `DELETE /discipleship/pairs/[id]`。
- ⚠ **缺口（G1）**：API 支持 `PATCH /discipleship/pairs/[id]`（如改 `status` 为
  `paused` / `completed`），但 UI **未提供配对的编辑 / 状态变更**入口。
- ⚠ **访问控制关注（G2）**：「🔗 表单」按钮与 `PairProgressModal` 的「复制链接 / 打开表单」
  对**所有登录角色（含只读）** 可见；公开表单端点无鉴权，因此只读用户可借总览中暴露的
  token 打开 `/d/[token]` 并提交每日进度，绕过「只读不可写」。见 §6。

### 3.8 用户管理 `/settings`（登录账户 · 权限角色，仅超级管理员）

- 导航项 `/settings` 带 `role: SuperAdmin` 门控；非超级管理员即使手动进入，页面主体显示
  「仅超级管理员可访问用户管理」，仅保留「修改我的密码」。服务端 `/accounts*` 读写均仅超级
  管理员。
- **C 新建账户**（「＋ 新建账户」）：关联成员（已有账户者不显示）、登录邮箱、权限角色
  （默认同工）、初始密码（≥ 8 位）。`POST /accounts`（明文 `password` 服务端哈希为
  `password_hash`）。
- **R**：权限说明卡（四角色能力列表）、账户表格（关联成员、在组身份、权限角色、邮箱、状态、
  最近登录）。
- **U 编辑账户**（点账户进入详情「保存账户设置」）：邮箱、权限角色、账户状态（启用 / 停用）、
  两步验证开关（规划中）、界面语言、三类通知开关。
- **U 重设密码**（详情「重设该账户密码」）：超级管理员为任意账户直接重设（无需旧密码，
  ≥ 8 位）`POST /accounts/[id]/password`。
- **D 删除账户**（详情「删除账户」）：确认「其成员档案会保留」，`DELETE /accounts/[id]`。
- ⚠ **小缺口（G2 UX）**：`usePageChrome` 无条件设置顶栏动作，非超级管理员误入 `/settings`
  时顶栏仍显示「＋ 新建账户」按钮（点击无效，因新建弹窗只在超管分支渲染）。见 §6。

### 3.9 登录 / 退出 / 改密 / 公开门训表单

- **登录 `/login`**：邮箱 + 密码（含显示 / 隐藏切换），`POST /api/auth/login`。账户不存在或
  停用 → `401 账户不存在或已停用`；密码错误 → `401 邮箱或密码错误`。成功写会话 Cookie 并
  更新 `last_sign_in_at`，跳转 `/`。
- **会话守卫**：`AppShell` 挂载即 `GET /api/auth/me`，失败重定向 `/login`。
- **退出登录**（侧栏账户菜单「↩ 退出登录」）：经 `useConfirm`（`danger`）确认后
  `POST /api/auth/logout` 清除 Cookie，跳 `/login`。
- **修改我的密码**（侧栏「🔑 修改我的密码」或用户管理页按钮，`ChangePasswordModal`）：当前
  密码 + 新密码 + 确认；`POST /api/auth/password`，服务端校验旧密码、新密码 ≥ 8 位、两次一致。
- **公开门训表单 `/d/[token]`**：无需登录。展示带领者 → 被带领者、40 天进度格、进度条；选择
  第几天、已完成 / 未完成、备注，提交 `POST /api/discipleship/form/[token]/progress`（upsert）。
  日期选择器默认定位到下一个未填天。提交后显示成功态，可「再填一天」。

## 4. 用户故事

> 每条含 2–4 条可测**验收标准**，覆盖正常路径与访问控制负例。角色简称：超管 = 超级管理员。

### 认证与会话

**US-01：作为教牧同工，我想用邮箱密码登录，以便进入管理后台。**
- 在 `/login` 输入有效邮箱 + 密码，点「登录」后跳转到 `/`（仪表盘）。
- 邮箱或密码错误时，卡片内显示「⚠️ 邮箱或密码错误」，停留在登录页。
- 账户被停用（status≠active）时提示「账户不存在或已停用」。
- 密码框「👁 / 🙈」可切换明文显示。

**US-02：作为未登录访客，我访问任意后台页应被拦截，以便保障数据安全。**
- 直接访问 `/members` 等页，`GET /api/auth/me` 401 后浏览器重定向到 `/login`。
- 任意受保护 API 无有效 Cookie 时返回 `401 未登录`。

**US-03：作为已登录用户，我想退出登录，以便在共享设备上保护账户。**
- 点侧栏账户区 →「↩ 退出登录」，弹出危险确认对话框「确定要退出当前账户吗？」。
- 确认后 Cookie 清除并跳转 `/login`；取消则留在原页。

**US-04：作为任意角色用户，我想自助修改我的密码，以便定期更换。**
- 侧栏「🔑 修改我的密码」打开弹窗，需填当前密码、新密码、确认新密码。

  `danger` 确认对话框；取消则不发生任何变更。无原生 `window.confirm`。

51. 用户管理非超管门控 · admin · 进入 `/settings` · 断言只读拦截文案 + 仅「修改我的密码」。

### 破坏性确认
52. 删除确认覆盖 · admin/super_admin · 逐一触发成员 / 小组 / 周聚会 / 聚会 / 奉献 / 课程 / 配对 / 账户删除与移出小组 · 断言每次都弹 `danger` 确认，取消不产生变更。

## 6. 审计发现（附录：CRUD 缺口与访问控制不一致）

按「安全 / 正确性 > CRUD 完整性 > 整洁度」排序：

1. **（G2）只读用户可经公开门训表单写入进度。** `/discipleship` 的「🔗 表单」按钮及
   `PairProgressModal` 的「复制链接 / 打开表单」对所有登录角色（含 readonly）可见；公开端点
   `/api/discipleship/form/[token]/*` 无鉴权。只读用户可借总览暴露的 `form_token` 打开
   `/d/[token]` 并提交每日进度，绕过「只读不可写」。失败场景：只读账户篡改门训进度而无审计。
   建议：对已登录会话访问表单端点时校验角色，或在只读会话下隐藏表单链接。

2. **（G2 · UX）只读用户可打开聚会「点名」并点击保存。** `/events` 的「点名」按钮与
   `AttendancePanel` 保存动作未按 `perms.write` 隐藏 / 禁用；只读点保存虽被服务端 403 拦截，
   但属「只会返回 403 的按钮」，违反 G2 客户端应隐藏无权操作的要求。建议：`perms.write` 才渲染
   「点名」或禁用其保存按钮。

3. **（G2 · UX）非超管误入 `/settings` 时顶栏仍显示「＋ 新建账户」。** `usePageChrome` 无条件
   设置动作按钮；虽点击无效（弹窗仅在超管分支渲染），仍是不该出现的入口。建议：在
   `isSuperAdmin` 为假时不设置该动作。

4. **（G1）培训场次缺 U / D。** API 有 `PATCH` / `DELETE /trainings/sessions/[id]`，UI 无
   编辑 / 删除场次入口。加错场次只能改数据库。

5. **（G1）培训报名状态流转不完整。** API 支持 `in_progress` / `completed`（自动补
   `completed_at`、`progress=100`）/ `dropped` 及 `DELETE /trainings/enrollments/[id]`；UI 仅
   `pending→approved`，无拒绝 / 移除报名 / 标记完成 / 退出。

6. **（G1）门训计划（programs）无 UI 管理。** 页面在无计划时要求「在数据库中创建
   discipleship_programs」；`GET/POST /discipleship/programs` 存在但页面未暴露创建 / 编辑 /
   删除计划的入口。

7. **（G1）门训配对缺 U。** API 有 `PATCH /discipleship/pairs/[id]`（可改 status 为 paused /
   completed），UI 无配对编辑 / 状态变更控件；配对状态实际只能停留在默认值。

8. **（G1 · 轻微）应用内无守望进度录入控件。** 进度只能经公开表单写入；牧者 / 管理员在应用
   内无法直接为某天打勾（须复制链接到表单页）。若这是刻意设计（仅带领者填表），应在 `docs/`
   明确记录。

9. **（轻微 · 文案）小组每周出席空态写「点『＋ 添加本周』」，实际按钮为「＋ 添加一周」。** 非功能
   缺陷，但违反 G8 用户可见字符串一致性。会议 `note` 字段 API 支持但 UI 从不采集。

10. **（轻微）成员牧师身份（church_role=pastor）无法在 UI 设定。** 新增仅置 `member`，编辑不含
    church_role；牧师身份只能经数据库写入。若为刻意（牧师身份由后台维护）应文档化。
